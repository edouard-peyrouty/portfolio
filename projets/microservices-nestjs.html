Architecture microservices avec NestJS

Application backend développée en TypeScript avec le framework NestJS, implémentant une architecture microservices composée de trois services indépendants : un service de gestion des utilisateurs, un service de gestion des commandes, et une API Gateway centralisant les requêtes entrantes. L'ensemble est conteneurisé avec Docker et sécurisé par authentification JWT.

Architecture microservices et API Gateway

Le projet repose sur trois services NestJS tournant en parallèle sur des ports distincts. L'API Gateway (port 3002) intercepte toutes les requêtes entrantes et les redirige via Axios vers le users-service (port 3000) ou l'orders-service (port 3001) selon l'URL, en conservant la méthode HTTP et le body original.


Chaque service suit une architecture Clean avec une séparation stricte des couches : les DTOs avec validation via class-validator dans adapter.in, la logique métier dans application, les entités dans domain et l'accès aux données dans infrastructure. L'injection de dépendances est gérée nativement par les modules NestJS.

Sécurité JWT

L'authentification repose sur des tokens JWT générés par le users-service lors de la connexion. Chaque service dispose d'un guard qui intercepte les requêtes entrantes, vérifie la présence et la validité du token dans le header Authorization, et renvoie une erreur 401 si le token est absent ou invalide.

Conteneurisation Docker

Chaque service est conteneurisé avec Docker, avec des Dockerfiles dédiés installant les dépendances et exposant les ports nécessaires. Un fichier docker-compose.yml orchestre le lancement simultané des trois services, simplifiant le développement et le déploiement.

Documentation Swagger

Chaque service expose une documentation interactive Swagger accessible à /api-docs, générée automatiquement depuis les décorateurs NestJS présents dans les DTOs et les controllers. Elle liste l'ensemble des endpoints avec leurs paramètres, types de réponses et codes HTTP possibles, et intègre l'authentification Bearer pour tester les routes protégées directement depuis l'interface.

Tests unitaires Jest et tests d'intégration Supertest

Les services sont couverts par des tests unitaires Jest ciblant la logique métier des services, avec des repositories mockés pour isoler les comportements à tester. Des tests d'intégration via Supertest viennent compléter la couverture en testant les endpoints REST réels, incluant l'authentification JWT et les règles métier comme l'interdiction de passer une commande à SHIPPED sans être préalablement à PAID. Un fichier de tests end-to-end passant par la gateway est également disponible, utilisant Axios pour simuler le parcours complet d'une requête.
